Analisador Léxico em Java para Mini Pascal

Um estudo simples e completo sobre a primeira fase de um compilador, explicado para estudantes, professores e pessoas sem experiência em programação.

Projeto: Editor e Analisador Léxico
Linguagem usada: Java
Interface: Java Swing

1. Início: Por que este projeto existe?

Antes de um computador executar um programa, ele precisa compreender o texto escrito pelo programador. Para uma pessoa, uma linha como idade := 18; pode parecer simples. Para o computador, porém, esse texto precisa ser dividido em partes menores e classificadas. Essa primeira divisão é chamada análise léxica.

Este projeto nasceu com a ideia de criar um editor simples, feito em Java, capaz de receber código inspirado em Mini Pascal e mostrar quais elementos existem dentro desse código. O sistema identifica palavras reservadas, identificadores, números, operadores, símbolos, comentários, textos e possíveis erros léxicos.

Significado central: o analisador léxico funciona como um leitor inicial do código. Ele não interpreta todo o programa, mas separa o texto em peças organizadas chamadas tokens.

2. O que é um token?

Um token é uma unidade reconhecida dentro de um programa. Podemos comparar com uma frase em português: antes de entender o significado completo da frase, precisamos reconhecer as palavras, sinais de pontuação e números. Na programação acontece algo semelhante.

Exemplo no código Tipo de token Significado
program Palavra reservada Palavra que já pertence à linguagem e tem função definida.
idade Identificador Nome criado pelo programador para variável, função ou outro elemento.
18 Número inteiro Valor numérico sem parte decimal.
15.75 Número real Valor numérico com parte decimal.
:= Atribuição Símbolo usado para guardar um valor em uma variável.

3. O que o analisador léxico faz?

O analisador léxico percorre o código caractere por caractere. Ele observa letras, números, espaços, símbolos e quebras de linha. Quando encontra um padrão conhecido, cria um token e guarda informações como:

  • o texto encontrado, chamado lexema;
  • a classe do token, como identificador ou palavra reservada;
  • o padrão reconhecido;
  • uma descrição compreensível;
  • a linha onde o token aparece.

Exemplo de entrada: 

program Escola;
var
  idade: integer;
begin
  idade := 18;
end.

Parte da saída esperada:

Lexema Classe Descrição
program PALAVRA_RESERVADA Palavra com significado fixo na linguagem.
Escola IDENTIFICADOR Nome definido pelo programador.
idade IDENTIFICADOR Nome definido pelo programador.
integer PALAVRA_RESERVADA Palavra com significado fixo na linguagem.
:= ATRIBUICAO Atribuição de valor.

4. Palavras reservadas

Palavras reservadas são palavras que pertencem à linguagem e não devem ser usadas livremente como nomes comuns. Elas representam comandos, tipos, estruturas de decisão, estruturas de repetição ou valores especiais.

O projeto reconhece palavras como: 

program, const, type, var, array, of, record, begin, end,
integer, real, boolean, char, string, if, then, else, while,
do, for, to, downto, repeat, until, read, readln, write,
writeln, div, mod, and, or, not, true, false, function, procedure

No editor, essas palavras aparecem coloridas para facilitar a leitura visual do código. Isso ajuda o aluno a perceber rapidamente quais termos pertencem à linguagem.

5. Identificadores

Identificadores são nomes criados pelo programador. Eles podem representar variáveis, funções, procedimentos ou outros elementos. No projeto, o padrão usado para identificadores é: 

letra (letra | digito | _)*

Isso significa que o nome deve começar com uma letra. Depois da primeira letra, pode conter letras, números ou sublinhado.

Identificador Válido? Motivo
idade Sim Começa com letra e contém apenas letras.
valor1 Sim Começa com letra e termina com número.
nome_aluno Sim Usa sublinhado após iniciar com letra.
1idade Não como identificador único Começa com número, por isso será separado em número e identificador.

6. Comentários

Comentários são textos escritos para explicar o código aos seres humanos. Eles não mudam o funcionamento do programa. O analisador reconhece três formas de comentário: 

{ comentario em Pascal }
(* outro comentario em Pascal *)
// comentario de uma linha

Reconhecer comentários é importante porque evita que o analisador trate palavras explicativas como comandos ou identificadores reais do programa.

7. Números, textos e símbolos

O sistema reconhece números inteiros, números reais, textos entre aspas simples, operadores aritméticos, operadores relacionais e símbolos de organização.

Categoria Exemplos
Número inteiro 0, 18, 2026
Número real 10.5, 15.75, 3.14
Texto 'Aluno maior de idade'
Operadores +, -, *, /, =, <, >=, <>
Símbolos ;, :, ,, ., (, ), [, ]

8. Erros léxicos

Um erro léxico acontece quando aparece no código um caractere ou uma sequência que o analisador não consegue reconhecer pelas regras definidas. Por exemplo, se a linguagem não aceitar o símbolo @, ele será sinalizado como inválido.

O projeto também detecta situações como:

  • comentário iniciado com { e não fechado com };
  • comentário iniciado com (* e não fechado com *);
  • texto iniciado com aspas simples e não fechado corretamente;
  • símbolos que não pertencem ao conjunto reconhecido.

9. Como a interface ajuda o estudante?

A interface foi construída para funcionar como um ambiente de aprendizagem. O estudante pode escrever código, executar a análise e observar imediatamente a tabela de tokens. Isso transforma um conceito abstrato em algo visual e manipulável.

  • O editor permite escrever ou abrir código.
  • As palavras reservadas são destacadas em cor diferente.
  • O botão de análise mostra os tokens em tabela.
  • A aba de erros mostra problemas encontrados no código.
  • O botão de exemplo carrega um programa pronto.
  • A exportação permite guardar os resultados para estudo ou entrega.

10. Impacto na sociedade escolar

O projeto pode impactar positivamente a sociedade escolar porque aproxima estudantes de conceitos fundamentais de compiladores e linguagens de programação. Em vez de estudar apenas teoria, o aluno pode ver na prática como o computador começa a ler um programa.

Para professores, a ferramenta pode servir como apoio em aulas de programação, lógica, compiladores ou introdução à ciência da computação. Para estudantes iniciantes, ela reduz a distância entre o texto do código e a estrutura interna que uma máquina precisa construir.

Valor educativo: o sistema não substitui o professor, mas torna visível uma etapa que normalmente fica escondida. Isso facilita explicações, demonstrações e exercícios.

11. Limitações atuais

É importante apresentar o projeto com honestidade técnica. Ele realiza análise léxica, mas ainda não realiza análise sintática nem análise semântica.

Fase Estado no projeto Explicação simples
Análise léxica Implementada Divide o código em tokens e encontra erros de símbolos ou sequências inválidas.
Análise sintática Não implementada Não verifica se os tokens estão na ordem correta segundo a gramática.
Análise semântica Não implementada Não verifica significado, tipos de dados, variáveis declaradas ou compatibilidade.

Por exemplo, o analisador pode reconhecer os tokens de uma linha incorreta, mas ainda não sabe dizer se a frase completa está gramaticalmente correta: 

var idade integer

Lexicamente, existem tokens reconhecíveis. Sintaticamente, porém, pode faltar o símbolo :. Essa verificação pertence a uma próxima fase.

12. Próximos passos possíveis

Para evoluir o projeto, os próximos passos naturais seriam:

  1. criar uma gramática simples de Mini Pascal;
  2. implementar um analisador sintático para verificar a ordem correta dos tokens;
  3. criar uma tabela de símbolos para guardar variáveis declaradas;
  4. implementar análise semântica para verificar tipos e uso de variáveis;
  5. gerar mensagens de erro mais próximas das mensagens de compiladores reais.

13. Fim: conclusão

Este projeto representa a primeira fase de um compilador: a análise léxica. Ele transforma código escrito em Mini Pascal em uma lista organizada de tokens, mostrando o lexema, a classe, o padrão, a descrição e a linha onde cada elemento aparece.

Mesmo não sendo ainda um compilador completo, o projeto tem valor pedagógico. Ele mostra como um computador começa a compreender código e ajuda estudantes a entenderem a diferença entre texto comum e estruturas reconhecidas por uma linguagem de programação.

O fim deste trabalho não deve ser visto como encerramento definitivo, mas como uma base. A partir da análise léxica, torna-se possível avançar para análise sintática, análise semântica e, futuramente, para um compilador ou interpretador mais completo.
